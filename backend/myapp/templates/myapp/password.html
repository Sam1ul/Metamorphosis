Password

Introduction

Passwords are the most common form of authentication. Good password handling is critical to protect user accounts and sensitive data. Weak handling (poor storage, weak policies, insecure reset flows) leads to account compromise, data breaches and lateral movement for attackers.

Types of Passwords / Credentials

  • User-chosen passwords: Entered by users (often weak or reused).
  • Generated passwords / passphrases: System- or user-generated, usually stronger.
  • API keys & tokens: Treated like passwords — must be stored and rotated securely.
  • One-time passwords (OTP): Short-lived codes (TOTP, SMS, email) used as second factors.
  • Recovery tokens / reset links: Sensitive; expire quickly and must be one-time use.

Common Attacks Against Passwords

  • Brute force: Try many combinations until one works (rate limiting and lockouts mitigate).
  • Dictionary / credential stuffing: Using leaked username/password pairs across sites.
  • Rainbow tables: Precomputed hashes used to reverse weak hashes (defeated by salts).
  • Phishing: Trick users into revealing credentials.
  • Keylogging / malware: Capture typed credentials on the client.
  • Password reset abuse: Exploit weak reset flows to take over accounts.

Secure Storage: Hashing & Salting

Never store passwords in plaintext. Use a strong, slow, memory-hard hash function and a unique per-password salt.

  • Salt: A unique random value per password appended or prepended before hashing. Prevents rainbow table attacks and ensures identical passwords have different hashes.
  • Hash: Use algorithms designed for passwords (arguably Argon2, bcrypt, scrypt, or PBKDF2 with adequate iterations).
  • Pepper (optional): A site-wide secret stored separately (e.g., in environment variables or HSM). If used, combine with salt before hashing. Pepper helps if DB is leaked but server config is intact.
  • Iterations / cost: Increase work factor to make hashing slow enough to deter attackers but acceptable for legitimate auth.

Recommended Algorithms

  • Argon2id: Current best practice — memory-hard, resistant to GPU attacks. Use tuned parameters for your environment.
  • bcrypt: Time-tested and safe when used with a sufficient cost factor (work factor).
  • scrypt: Memory-hard alternative.
  • PBKDF2-HMAC-SHA256: Acceptable if iterated sufficiently, but slower growth vs Argon2 for resisting GPU/FPGAs.

Important: Don’t invent your own scheme. Use well-reviewed libraries.

Password Policies & Strength

Password policies should balance security and usability. Avoid overly strict rules that push users to insecure workarounds (like writing passwords down).

  • Encourage length over complexity: Passphrases (4+ words) are often stronger and easier to remember than complex short passwords.
  • Minimums: Require a reasonable minimum length (e.g., 12 characters for user-chosen passwords) rather than brittle composition rules.
  • Disallow known-bad passwords: Check against breached-password lists (e.g., Have I Been Pwned API) and block commonly used passwords.
  • Rate limiting and lockouts: Add progressive delays or temporary lockouts after repeated failed attempts (avoid permanent lockouts that aid DoS).
  • Encourage password managers: Suggest using password managers to generate and store unique credentials.

Multi-Factor Authentication (MFA)

MFA dramatically reduces the risk from stolen or reused passwords. Offer multiple, secure second factors.

  • Recommended: Time-based one-time passwords (TOTP) via authenticator apps or hardware tokens (FIDO2/WebAuthn, U2F).
  • Acceptable with caution: SMS-based OTP — better than nothing, but vulnerable to SIM swap attacks; prefer authenticator apps/hardware keys.
  • Backup codes: One-time-use backup codes should be provided for account recovery and stored/displayed once.
  • Adaptive MFA: Consider prompting MFA on risky events (new device, new IP, high-value actions).

Password Recovery & Reset Flows

Recovery flows are frequent attack targets — design them to minimize abuse.

  • Reset link: Send a cryptographically random, single-use token (long, unguessable) via email with a short expiry (e.g., 15–60 minutes).
  • Token storage: Store hashed reset tokens server-side (so a leaked DB doesn’t expose tokens directly).
  • Rate-limit reset requests: Prevent email spamming or probing accounts.
  • Invalidate sessions: After a password reset, invalidate active sessions or require re-login.
  • Recovery options: Avoid insecure recovery questions (mother’s maiden name). Prefer email, MFA, or in-person/account verification flows.

UX & Developer Considerations

  • Show password strength meters: Use entropy-based meters that encourage longer passphrases rather than arbitrary complexity rules.
  • Inline validation: Validate on the client for format/length but always re-check on the server.
  • Clipboard & paste: Allow paste (disallowing paste encourages weak passwords) but warn users for very short inputs.
  • Account lockouts: Use progressive delays and captchas as alternatives to harsh permanent lockouts.
  • Password change confirmation: Email users when sensitive changes occur (password/email/MFA settings), include details of the change and how to revoke.
  • Logging & monitoring: Log failed login attempts, successful logins from new locations/devices, and reset attempts; monitor for unusual spikes (credential stuffing).

Best Practices (Summary)

  • Never store plaintext passwords.
  • Always use a strong password hashing algorithm with a per-password salt (Argon2id preferred).
  • Encourage/require MFA for sensitive accounts.
  • Block breached and common passwords.
  • Design secure reset flows (single-use tokens, short expiry, hashed storage).
  • Use rate limiting and monitoring to detect credential stuffing and brute force.
  • Promote password managers and passphrases for users.

Code Examples

Python — Argon2 (using argon2-cffi)

from argon2 import PasswordHasher

ph = PasswordHasher(time_cost=2, memory_cost=102400, parallelism=8)

# Hashing
hash = ph.hash("correct horse battery staple")

# Verifying
try:
    ph.verify(hash, "correct horse battery staple")
    # If verify passes, optionally check if parameters need rehashing:
    if ph.check_needs_rehash(hash):
        new_hash = ph.hash("correct horse battery staple")
except Exception:
    # handle invalid password
    pass

Python — bcrypt

import bcrypt

password = b"hunter2"
salt = bcrypt.gensalt(rounds=12)
hashed = bcrypt.hashpw(password, salt)

# Verify
bcrypt.checkpw(password, hashed)

Server-side: Generate a password reset token (recommend storing hashed token)

import secrets, hashlib

token = secrets.token_urlsafe(48)  # random token for link
hashed_token = hashlib.sha256(token.encode()).hexdigest()
# Store hashed_token in DB with expiry; send the raw token in reset URL.

Quick Checklist

  • Passwords hashed using Argon2 / bcrypt / scrypt with unique salts.
  • Strong password policy (length/passphrase + breached-password checks).
  • MFA available and encouraged/required for high-risk actions.
  • Secure reset flow using single-use, time-limited tokens.
  • Rate limiting, logging, and alerting around auth endpoints.
  • Password managers recommended to users.
  • Session invalidation after password changes.

Closing Summary

Password security is a combination of correct storage, smart policy, good UX, and strong incident handling. Implement modern hashing, use salts (and optionally a pepper), require or offer MFA, and design secure recovery flows — those combined measures dramatically reduce the risk of account compromise.

"Passwords are the first line of defense — make that line strong and resilient."
