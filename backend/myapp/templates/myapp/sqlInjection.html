SQL Injection

What is SQL Injection?

SQL Injection (SQLi) is a critical web security vulnerability that allows an attacker to interfere with the queries an application makes to its database. It happens when user input is inserted into an SQL statement without proper validation or sanitization.

  • Occurs when user input is treated as executable SQL code.
  • Allows attackers to manipulate database queries and access sensitive data.
  • Can affect both relational (MySQL, PostgreSQL) and non-relational systems if poorly handled.

Why It’s Dangerous

SQLi can compromise entire systems because it gives unauthorized access to the database. Attackers can bypass authentication, extract or delete sensitive data, and even take administrative control of the application.

  • Data theft — credentials, personal info, financial data.
  • Data corruption or deletion.
  • Full system compromise or privilege escalation.
  • Legal penalties under data protection laws (GDPR, HIPAA, etc.).

Types of SQL Injection

  • Classic / In-band SQLi: Attacker directly retrieves data using the same communication channel (e.g., UNION-based or Error-based).
  • Blind SQLi: Data is extracted by observing application behavior — no visible errors.
  • Time-based Blind SQLi: Uses deliberate delays to infer true/false conditions.
  • Out-of-Band SQLi: Uses alternate channels (like HTTP requests or DNS) to extract data.

Defensive Principles (Checklist)

  1. Use parameterized queries / prepared statements.
  2. Use ORM frameworks (Django ORM, SQLAlchemy, Hibernate) to separate logic from data.
  3. Apply least privilege — restrict database user permissions.
  4. Validate and sanitize inputs using allowlists for IDs, dates, enums, etc.
  5. Avoid dynamic SQL that concatenates user input directly.
  6. Escape inputs safely only if absolutely necessary using DB-safe methods.
  7. Encrypt sensitive data (at rest & in transit).
  8. Enable query logging and monitor unusual activity.
  9. Perform regular security testing (SAST, DAST, and penetration testing).
  10. Use WAF and rate limiting for extra protection.

How to Detect or Test (Responsibly)

  • Perform code reviews for SQL string concatenations.
  • Run SAST (Static Analysis Security Testing) tools in CI/CD pipelines.
  • Use DAST tools (like OWASP ZAP, Burp Suite) for authorized tests only.
  • Check for error messages or anomalies in query execution logs.
  • Never test or attack systems you do not own or have permission to test.

SQL Injection Techniques — Awareness & Defenses

String / Quote Injection

Attackers supply characters (quotes, backslashes, etc.) that change how string literals are delimited in SQL, potentially breaking the intended query structure.

Defense: use parameterized queries / prepared statements so user input is treated as data, never as SQL syntax.

Comment Injection

Use of comment syntax (for example --, #, or /* ... */) to skip or cut off the remainder of a query or condition.

Defense: parameterize queries and avoid building SQL strings from raw input; also validate and canonicalize input.

Boolean Logic Manipulation

Injecting additional logical expressions (OR, AND variants) to force WHERE clauses to evaluate differently, often attempting to bypass checks.

Defense: always bind parameters and validate input types/values before the DB layer.

Union / Concatenation Attempts

Attempts to merge or append result sets (via UNION or similar constructs) to retrieve data from other tables or queries.

Defense: minimize returned columns/rows, use parameterization, and avoid dynamic SQL that accepts arbitrary fragments.

Stacked Queries

Trying to terminate one statement and start another (depends on DB/driver support). Some drivers block multiple statements by default.

Defense: disable multi-statement execution in DB driver/config when possible and use parameterized single-statement APIs.

Subquery Injection

Injecting nested queries (e.g., (SELECT ...)) or subselects to alter the returned value or logic of a surrounding query.

Defense: use strong input validation, least-privilege accounts, and parameter binding to prevent user input from becoming SQL fragments.

Function / Builtin Abuse

Attempting to call database built-in functions (file, network, or OS-related functions) that could read files, generate outbound connections, or escalate impact.

Defense: harden DB configuration (disable unnecessary features), run app accounts with minimal privileges, and monitor suspicious function calls.

Blind Techniques (Boolean / Timing)

When the application doesn't return database output, attackers may infer information by observing true/false responses or timing differences.

Defense: avoid revealing internal state in responses, validate inputs, and monitor for repeated probing patterns or timing anomalies.

Encoding / Obfuscation

Payloads can be encoded (hex, unicode, URL encoding, etc.) or obfuscated to bypass naive filters that look for literal keywords or characters.

Defense: normalize and decode inputs server-side before validation; rely on parameterization rather than blacklists.

Parameter Pollution

Sending multiple values for the same parameter name (e.g., repeated query params) to confuse application logic or cause unexpected behavior.

Defense: define and enforce a clear parameter handling policy (first/last/combined) and validate canonicalized values before DB use.

Reminder: only use awareness information for defense, auditing, and authorized testing.
